Context Engineering for Agents

A practical breakdown of the four strategies — write, select, compress, and isolate — for filling an agent's context window with exactly the right information at each step. Source: blog.langchain.com · The LangChain Team · July 2, 2025

01What Is Context Engineering

As Andrej Karpathy frames it, LLMs are a new kind of operating system: the model is the CPU and its context window is RAM — limited working memory that must be carefully managed. Context engineering is the discipline of curating what goes into that window at every step of an agent's trajectory.

"Context engineering is the delicate art and science of filling the context window with just the right information for the next step." — Andrej Karpathy

Three categories of context must be managed: Instructions (prompts, memories, few-shot examples, tool descriptions), Knowledge (facts, retrieved documents), and Tools (feedback from tool calls). Cognition has called context engineering "effectively the #1 job of engineers building AI agents."

02The Four Strategies

Long-running agents accumulate token-heavy tool call histories, risking context overflow, inflated costs, and degraded performance. Drew Breunig identified four specific failure modes: Context Poisoning (hallucination enters the context), Context Distraction (context overrides training), Context Confusion (superfluous context skews response), and Context Clash (contradictory context segments).

These failure modes map to four mitigating strategies:

WWrite

Save context outside the window — scratchpads and long-term memories — so it is available without occupying tokens.

SSelect

Pull only the relevant context into the window — via RAG, embedding search, or explicit state field exposure.

CCompress

Retain only what is needed — summarise long trajectories, prune old messages, post-process verbose tool outputs.

IIsolate

Split context across sub-agents, sandboxed environments, or state fields so each unit sees only its relevant slice.

03Write: Scratchpads & Memory

Scratchpads persist information while an agent is performing a task. Anthropic's multi-agent researcher uses a Memory tool so the LeadResearcher saves its plan before the context window can be truncated at 200,000 tokens. Scratchpads can be implemented as a file-system tool call, or as a field in the agent's runtime state object — giving fine-grained control over what is exposed to the LLM at later turns.

Long-term memories persist across many sessions. Reflexion introduced self-generated reflection memories; Generative Agents synthesised memories periodically from past feedback. These patterns now appear in ChatGPT, Cursor, and Windsurf, all of which auto-generate long-term memories from user-agent interactions.

04Select: Retrieval & RAG

Selecting context means pulling the right information into the window. For procedural memories, many code agents use always-loaded files — Claude Code uses CLAUDE.md, Cursor uses rules files. For larger memory collections, agents rely on embedding search or knowledge graphs.

Selection can go wrong. Simon Willison described a case where ChatGPT unexpectedly fetched location from memories and injected it into an image request, making the user feel the context window "no longer belongs to them."

Tool selection is a related challenge: providing too many tools causes model confusion due to overlapping descriptions. Applying RAG over tool descriptions has been shown to improve tool selection accuracy by 3x. For knowledge retrieval at scale, code agents like Windsurf combine AST parsing, embedding search, grep, knowledge graph retrieval, and a re-ranking step, since embedding search alone becomes unreliable as codebases grow.

Context Type Selection Mechanism Production Example
Procedural instructions Always-loaded rules file CLAUDE.md, Cursor rules
Episodic memories Embedding similarity search ChatGPT memory, LangGraph store
Tool descriptions RAG over tool docs LangGraph Bigtool
Codebase knowledge AST + embedding + graph + rerank Windsurf, Cursor

05Compress & Isolate

Compressing context retains only the tokens required. Claude Code's auto-compact summarises the full agent trajectory once the context reaches 95% capacity. Summarisation can also be applied at specific design points — post-processing verbose tool calls, or at agent-agent handoff boundaries to reduce tokens during knowledge transfer. Trimming uses heuristics to prune older messages without LLM involvement.

Isolating context splits it across distinct boundaries. In multi-agent architectures (inspired by OpenAI Swarm), each sub-agent has its own context window, focused on a narrow sub-task. Anthropic's multi-agent researcher found that many agents with isolated contexts outperformed a single agent — at the cost of up to 15× more tokens.

Two additional isolation techniques: code sandboxes (HuggingFace's CodeAgent runs tool calls in an E2B sandbox; only selected return values re-enter the LLM context) and state schema isolation (LangGraph stores tool outputs in state fields not exposed to the LLM until explicitly needed).

06LangGraph Implementation

LangGraph is designed around a typed state object with a schema. Each node (step) reads from and writes to state, giving precise control over what context is exposed to the LLM at each agent turn. This single primitive enables all four strategies.

For writing, LangGraph's checkpointing persists agent state across all steps (short-term scratchpad) and LangMem provides long-term memory abstractions for cross-session persistence. For selecting, nodes access state directly, and LangGraph's memory store supports embedding-based retrieval for large collections. LangGraph Bigtool applies semantic search over tool descriptions for large tool sets.

For compressing, built-in utilities support message summarisation and trimming; additional logic can post-process specific tool calls or add summarisation nodes at key agent phases. For isolating, LangGraph supports multi-agent architectures via the supervisor and swarm libraries, state schema isolation (fields not exposed to the LLM), and E2B/Pyodide sandboxes for code-executing agents.

Before applying any context strategy, two foundations are critical: use LangSmith to trace token usage and identify where context engineering will have the most impact; then establish agent evaluation baselines so you can measure whether a change helps or hurts performance.
