01三層架構總覽

Kubernetes 的整體架構由三個層次組成：Control Plane（叢集大腦，負責決策與狀態管理）、Node（工作節點，負責實際執行容器）、以及 Pod（最小部署單位，封裝一或多個容器）。這三層各司其職，透過 API Server 作為統一通訊中心串連在一起。

此架構的核心設計哲學是「宣告式管理」——使用者描述期望狀態，Control Plane 持續協調實際狀態以趨近期望狀態，而不是直接下命令控制每個元件。

02Control Plane 核心元件

Control Plane 是 Kubernetes 叢集的大腦，包含四個主要元件，各自負責不同的決策與儲存職責：

AAPI Server

叢集的統一入口，所有 kubectl 指令、元件間通訊都透過 kube-apiserver 進行。負責驗證、授權與狀態持久化。

SScheduler

監聽未排程的 Pod，根據資源需求、親和性規則、節點狀態，決定將 Pod 派送至哪個 Node 執行。

Eetcd

分散式鍵值資料庫，是 Kubernetes 所有叢集狀態的唯一真實來源（Single Source of Truth）。生產環境需要高可用備份。

CController Manager

執行各類控制迴圈（Control Loop），持續比對實際狀態與期望狀態，驅動系統趨近目標，例如 ReplicaSet 控制器確保 Pod 數量正確。

宣告式協調原則：Controller Manager 的每個控制迴圈持續監聽 etcd 的狀態變化，當實際狀態偏離期望狀態時自動修正，這是 Kubernetes 自癒能力的核心機制。

03Node 層元件

每個 Worker Node 上運行三個關鍵代理程式，負責接收 Control Plane 的指令並在本機執行容器工作負載：

元件 職責 通訊對象
kubelet Node 的主代理，負責啟動、停止、監控 Pod；定期向 API Server 回報節點健康狀態 API Server、Container Runtime
kube-proxy 維護節點上的網路規則（iptables / IPVS），實現 Service 的 ClusterIP 流量轉發與負載均衡 API Server（監聽 Service 變更）
Container Runtime 實際執行容器映像的引擎（containerd、CRI-O 等），實作 CRI 介面供 kubelet 呼叫 kubelet（CRI 協定）

kubelet 透過 CRI（Container Runtime Interface）與容器執行環境解耦，使得 Kubernetes 可以支援多種容器運行時，不再依賴單一實作。

04Pod：最小部署單位

Pod 是 Kubernetes 中可排程的最小單元，封裝一或多個緊密相關的容器，這些容器共享同一個網路命名空間（相同 IP）和儲存卷（Volume）。

直接建立 Pod（naked pod）並不建議用於生產環境——Pod 本身不具備自我修復能力，節點故障後不會自動重建。實際部署應透過 Deployment、StatefulSet 等上層物件管理 Pod 的生命週期。

設計建議：一個 Pod 內通常只放一個主容器，Sidecar 模式（如日誌收集、服務網格代理）是多容器 Pod 的主要使用場景。避免將不相關的服務放入同一個 Pod。

05核心工作負載物件

Kubernetes 提供多種高階物件管理 Pod 生命週期。最常用的 Deployment 管理無狀態服務，透過 ReplicaSet 確保指定數量的 Pod 副本持續運行；Service 為一組 Pod 提供穩定的虛擬 IP（ClusterIP）端點，即使 Pod 因重啟而 IP 改變，Service 的 endpoint 依然不變。

ConfigMap 與 Secret 負責設定注入：ConfigMap 儲存明文設定（環境變數、設定檔），Secret 以 base64 編碼儲存敏感資訊（密碼、Token），兩者皆可掛載為環境變數或 Volume。

06網路與流量入口

Ingress 作為 L7（HTTP/HTTPS）負載均衡入口，根據 Host 或 Path 規則將外部流量路由至叢集內部不同的 Service。Ingress 本身是路由規則的宣告，需要搭配 Ingress Controller（如 Nginx、Traefik）才能實際執行。

相較於 NodePort 或 LoadBalancer 類型的 Service，Ingress 可在單一入口點集中管理 TLS 終止、虛擬主機路由與 URL rewrite，大幅降低對外暴露的端點數量。

07自動擴縮與設定注入

HPA（Horizontal Pod Autoscaler） 根據 CPU 使用率或自訂 memory metrics，自動調整 Deployment 的副本數（replicas），使叢集在流量高峰時自動橫向擴展、閒置時縮減，達到資源利用率最佳化。

設定注入方面，ConfigMap 儲存非敏感設定（如 feature flag、資料庫連線字串），Secret 儲存敏感資料（如 API key、TLS 憑證），兩者皆可透過環境變數或 Volume 掛載方式注入 Pod，避免在容器映像中硬編碼設定。

物件 用途 儲存方式 注入方式
HPA 依 metrics 自動調整 Pod 副本數 叢集資源物件 自動修改 Deployment replicas
ConfigMap 非敏感設定注入 明文 key-value env / Volume / envFrom
Secret 敏感資料注入 base64 編碼 env / Volume / imagePullSecret
HPA 建議：設定 HPA 時需同時設定 minReplicas 與 maxReplicas，並確保 Pod 的 resource requests 已正確設定，否則 HPA 無法計算 CPU 使用率百分比。
