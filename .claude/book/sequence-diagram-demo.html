01 什麼是 sequence diagram

Sequence diagram 不是流程圖。它強調的是時間順序 + 跨 actor 互動：時間軸從上往下，每條垂直 lifeline 代表一個 actor 或 participant，水平 arrow 是訊息。同步呼叫用實線實心箭頭；回傳用虛線。Actor 在處理工作時，lifeline 上會疊一條細長的 activation bar 標記「現在在 busy」。Actor 對自己呼叫的內部處理用 self-loop（一段右出 → 下 → 左回的折線）。

用得對的場景：3 個以上 actor、有 round-trip、互動順序本身就是要傳達的重點。例如 OAuth flow、API 編排、microservices 對話、CI/CD pipeline 內外部呼叫。不適合：純粹的決策樹（用 flowchart）、靜態系統結構（用 architecture）。

03 跟 flowchart 的分工

很容易混淆。一個簡單的判讀方法：多 actor + 時序往下走 = sequence，單一系統 + 決策分支 = flowchart。

軸SequenceFlowchart
主軸時間順序邏輯分支
節點actor / participantstep / decision
連線messages（請求 / 回傳）flow（yes / no）
適合OAuth、API 編排、微服務、CI/CDtoken resolver、driver dispatch、parser state machine
不適合單一系統內的 if-else 樹3+ actor 互動 round-trip
前一份 book（book diagram port v1）用 flowchart 畫 token 三層 fallback — 因為那是單一系統內的決策樹。這份 book 改用 sequence 畫 /book pipeline — 因為那是多 actor round-trip。兩種圖各司其職，混用會視覺記憶錯位。
