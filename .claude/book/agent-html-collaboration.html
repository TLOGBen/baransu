Agent 協作方式

以 HTML 取代 Markdown

當 agent 的輸出越來越長、越來越複雜，Markdown 已無法承載所需的資訊密度。 HTML 不只是更好看的格式——它重新定義了人類與 agent 的協作方式。 來源：@trq212 · 2026-05-08 · via Claude Code

01Markdown 已成為 Agent 溝通的瓶頸

Markdown 自誕生以來一直是 agent 與人類溝通的預設語言，語法簡單、格式可攜。 然而，隨著 agent 的能力不斷擴張，這個默契開始出現裂縫。

Agent 產出的文件越來越長，百行以上的 Markdown 讓人難以消化。 更關鍵的是，人們不再親手修改這些文件，而是把修改工作也交還給 agent， Markdown「易於人工編輯」的核心優勢因此蒸發。

最明顯的症狀：agent 被迫用 ASCII 字元繪製圖表、用 Unicode 符號近似顏色—— 這些奇特的做法不是 agent 的選擇，而是在有限格式下盡力填補表達缺口的結果。

02HTML 作為新的協作媒介：核心優勢

如果說 Markdown 是一張白紙，HTML 就是一整座工作室。

HTML 文件可以內嵌滑桿、選項鈕、即時預覽，讓使用者在不寫程式碼的情況下調整參數， 並透過「複製為 prompt」按鈕將調整結果直接貼回 Claude Code。

03HTML 放大 Agent 的上下文攝取能力

HTML 格式的另一個優勢不在於輸出本身，而在於它如何與 agent 的上下文攝取能力相乘。 Claude Code 可以讀取整個程式碼資料夾、透過 MCP 連接 Slack、Linear、瀏覽器歷史、 git log 等外部資料源，再將這些分散的資訊整合進一份 HTML 文件。

當驗證 agent 需要確認某項實作是否符合規格時，一份 HTML 規格文件（包含設計稿、資料流圖、關鍵程式碼片段），遠比零散的 Markdown 片段更能建立完整的上下文——更豐富的上下文意味著更準確的驗收。

04具體協作場景

🗺 規劃與探索

六種並排設計方案、標注取捨的比較表。方向確認後新開 session，以 HTML 作為規格輸入。

🔍 程式碼審查

渲染帶行內標注的 diff、色彩標記問題清單。HTML code explainer 附於每個 PR。

🎨 設計與原型

內嵌動畫、滑桿調參、即時預覽，全在同一份文件裡完成，確認後轉譯成 React / Swift。

📊 報告與研究

Slack + codebase + git 歷史整合為互動式 HTML，以分頁、SVG 圖表、可折疊段落組織。

⚙️ 自訂編輯介面

為單次任務打造一次性 HTML 編輯器（拖曳票券、feature flag 表單、Prompt 模板預覽）。 關鍵：「複製為 JSON/prompt」匯出按鈕，將 UI 決策轉化回 agent 可接收的格式。

生成時間： HTML 通常需要 Markdown 的 2–4 倍時間。但若更豐富的格式讓人類真正去讀、真正做出決策，token 成本換來的資訊利用率提升在整體協作效率上是正收益。

入門門檻遠比想像中低——直接要求「make a HTML file」即可。 設計風格一致性問題可以透過讓 agent 讀取既有 codebase 生成設計系統 HTML， 作為後續所有文件的視覺參考基準來解決。

06「保持在迴路中」——協作的本質動機

採用 HTML 最根本的動機，是把決策權從 agent 手裡拿回來。 當 agent 輸出的文件長達數百行 Markdown，使用者漸漸停止真正閱讀它們， 默默將決策權讓渡給 agent。這種「盲目信任」對於需要人類判斷的決策點而言是一種風險。

HTML 的豐富視覺化讓使用者重新投入閱讀——不是被迫閱讀， 而是因為文件本身足夠清晰、足夠有趣而主動閱讀。 這讓使用者得以在 agent 工作流的關鍵節點真正理解正在發生的事， 並在需要時做出有意識的介入。

「保持在迴路中」不只是情感上的滿足，更是高品質人機協作的必要條件。
