01共同哲學底座

兩份 skill 雖然定位不同——guizang 專做網頁 PPT，huashu 涵蓋原型／動畫／簡報／評審——但對「為什麼 AI 默認的設計醜」這件事，給出了驚人一致的答案：AI 默認產出 = 訓練語料的視覺平均 = 沒有任何品牌被認出來。要繞開這個陷阱，靠的不是更好的審美，而是「在生成之前先收窄選擇空間」。

從這個共識出發，兩份 skill 都同意三件事。第一，凭空做 hi-fi 是 last resort，一定要先找到「上下文」——guizang 是七問澄清裡的素材／圖片／主題色，huashu 是 Core Asset Protocol 的 logo／產品圖／UI／色值五件套。第二，沒有上下文就不要硬畫，宁可停下問用戶，或用「誠實 placeholder」（灰塊 + 文字標籤）佔位，不要用 CSS 剪影、SVG 手畫、generic 渐变來凑。第三，把所有臨場決策做成檔案——guizang 寫進 `項目記錄.md` / `大綱-v1.md`，huashu 寫進 `brand-spec.md` / `product-facts.md`——agent 自己不能信任自己的記憶。

共同根 axiom：視覺品質的瓶頸不是「能不能畫得好看」，而是「能不能避開所有 AI 默認模式」。反 slop 不是審美洁癖，是替用戶保護品牌識別度。

02guizang 的理念：單一場景的硬鎖死

guizang 把整個 skill 收斂在一件事上——做網頁 PPT。能收這麼緊，是因為它選擇了「以原始 PPT 為 Golden Source」的策略：`swiss-layout-lock.md` 直接寫上作者本機路徑 `/Users/guohao/Documents/op7418的仓库/项目/Thin-Harness-Fat-Skills/ppt/index.html`，22 個 Swiss 版式都從這份原始檔反推，不允許在 LLM 端發明新版式。

這套理念把「視覺品質」轉成「結構服從」問題。每個 `

` 必須標 `data-layout="Sxx"`；每個版式表寫明「必須保留的骨架 / 圖片規則 / 圖片比例」；配套的 `scripts/validate-swiss-deck.mjs` 機器校驗會攔截未登記版式、SVG 內含可見文字、`object-position:top center` 用於照片等違規。`checklist.md` 524 行則沉澱了三層 P0/P1/P2 質檢項，每條都有「現象 → 根因 → 做法」三欄結構——0-A 那條畫布對齊法則甚至精確到「`.canvas-card` 已自帶 padding，主體區再寫 padding 就會內縮重複 5vw」。

主題色也走「整套替換」協議：`themes-swiss.md` 規定每套配色都遵循「高級灰白底 + 單一高飽和高亮色」（IKB / 檸黃 / 檸綠 / 安全橙），明文禁止混搭多個 accent；換主題只動 `:root{}` 區塊裡的十個 CSS variables（`--paper / --ink / --accent / ...`），其他 CSS 都走 `var(--...)`，零額外改動。

03huashu 的理念：多場景 + 哲學注入

huashu 走相反方向——一個 skill 蓋住原型／動畫／簡報／變體／評審五大場景，靠的是把「哲學」寫進 SKILL.md 主體，作為決策樹的優先級。最上層是 Principle #0 Fact Verification First：凌駕所有其他流程，提到具體產品/技術必先 `WebSearch` 驗證版本與規格，禁止句式「我記得 X 還沒發布」「據我所知 X 的規格是...」明文列出。這條原則是 2026-04-20 的真實事故沉澱：作者凭記憶以為 DJI Pocket 4 沒發布，做了概念剪影動畫，結果產品 4 天前剛發布。

第二層是 核心資產協議（v1.1 從「品牌資產」升級）：5 步硬流程 ask → search → download → verify → freeze，每步都有 fallback。協議的核心理念是「資產 > 規範」——品牌的本質是被認出來，logo / 產品圖 / UI 截圖的識別度貢獻遠大於色值與字體。協議另含「5-10-2-8」素材鐵律：5 輪搜索 / 10 個候選 / 選 2 個好的 / 每個 ≥8/10 分。作者揭露 A/B 測試結果：v2 協議讓 6 個 agent 的穩定性方差降 5 倍。

再往下是 Junior Designer 模式——先寫 assumption + reasoning + placeholders 進 HTML、早期 show 給用戶、確認方向後再填真內容、再 show、再迭代細節。原則一句話：「理解錯了早改比晚改便宜 100 倍」。

差異化視角：guizang 用「鎖死版式」減少選擇；huashu 用「優先級哲學」收窄判斷。前者適合單一場景做深，後者適合多場景做廣。

04共通的處理手法

剝掉差異看共通：兩份 skill 在五個層面用了幾乎一樣的結構模式。

檔案組織都是「`SKILL.md` 路由 + `references/*.md` 深度規則 + `scripts/*` 工具 + `assets/*` 模板」四層分離。SKILL.md 不放執行細節，靠 references 拆檔避免 context bloat：guizang 8 檔（layouts / themes / components / image-prompts / checklist / swiss-layout-lock）、huashu 21 檔（簡報 / 動畫 / 影音 / 工作流 / 評審五大類）。

動手前先閘門——guizang 是 Step 1 七問澄清（風格、受眾、時長、素材、圖片、主題色、硬約束），huashu 是 workflow.md 五類必問（Design Context / Variations / Fidelity & Scope / Tweaks / 問題專屬 4+ 條）。兩者都強調「一次列完讓用戶批量答」，不要一來一回問。

圖片不能憑生成式直覺處理——guizang 的「先選槽位再生成圖片」、huashu 的「Core Asset Protocol」、雙方共用的「禁止 CSS 剪影代替真實產品圖」鐵律，都把圖片視為比文字更敏感的議題。提示詞都有固定的負面尾巴：no title, no footer, no page chrome, no logo, no border。

生成後一定要驗——guizang 的 `validate-swiss-deck.mjs` 跟 huashu 的 `verify.py`（Playwright 控制台錯誤 + 截圖 + 多視口）方向不同，但都把「自動驗證」當成必經閘門，不允許「生成完就交付」。

把規則寫成檔案而非提示詞——兩份 skill 都把「過去踩過的坑」沉澱成 markdown，下次 agent 跑同一條路就會被檔案約束。這是兩者唯一的「自我演化機制」。

06給 /book 的啟示

對照下來，`/book` 的 PPT 不好看，可以拆成四個結構性缺口，每個都有對應的補丁來源。

1版式登記制

當前 §6 只 5 個版型，未鎖定。借 guizang `data-layout="Sxx"` + validator，把 `references/slide-template.html` 擴成 10+ layout catalog。

2配圖規格化

當前 PPT pipeline 對配圖零規範。新開 `references/slide-image-prompts.md`，定每版型的比例 / 提示詞模板 / object-position 規則。

3P0 checklist

當前 validator 只查元素是否存在。新開 `references/slide-checklist.md`，沉澱每次踩坑成「現象 / 根因 / 做法」三欄。

4編輯版 PPTX

當前 html2pptx 沒驗 4 條硬約束。借 huashu 寫法：DIV 不裝文字、不用渐变、background 與 text frame 分層、不用 background-image。

優先序：版式登記 → 配圖規格 → P0 checklist → 編輯版 PPTX。前兩項是結構性硬約束，做完馬上能看到輸出品質提升；後兩項是抓長尾與差異化，可分批迭代。

最終一句：guizang 教我們「視覺品質 = 結構服從」，huashu 教我們「結構服從之前要先選對結構」。對 /book 來說，這兩個層次要一起補才能往前走。
