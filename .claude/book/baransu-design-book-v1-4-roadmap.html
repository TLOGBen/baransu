01起點與目標

baransu plugin v1.3 已 ship 到 origin/main（commit 7d6eb15）；本 roadmap 為 v1.4 規劃。目標**不是抽象 feature parity**，而是「用 baransu /design + /book 能產出三 baseline 專案的等價 artifact 品質」。

三 baseline 對應產出能力

guizang-ppt-skillSwiss 國際主義風 PPT slide HTML。22 layouts (S01–S22) + 圖片協議 + 七問澄清前置。
huashu-designEditorial 印刷學 + 多場景（簡報 / 動畫 / 影音 / 工作流 / 評審 5 類，21 references files）。
tw93/Kami印刷品設計系統。8 文件類型（One-Pager / Long Doc / Letter / Portfolio / Resume / Slides / Equity Report / Changelog）+ 14 inline SVG 圖表 + 紙質感工藝。
基線雙軌平均機械骨架 ~52% + 視覺工藝 ~47% → 加權綜合 ~50%。落差 40 pp 主要在工藝層（typography 微觀、color craft、印刷學詞彙、AI slop 反向）。

「不像 AI 做的」識別度是核心 KPI。機械骨架（validator / GATE-F / pipeline）已 production-grade，但缺工藝細節導致輸出仍像「合規但平庸的 AI 產出」。v1.4 目標是補上落差。

05達不到的能力 — 與為何不可以

~95% 是 aspirational ceiling；剩下 5% 是 baransu 作為 Claude Code skill plugin 中的設計子模組本質限制。下列能力不在 v1.x scope 內，**不該強行追平**。

缺什麼能力 對應 baseline 能力 為何不可以
原生 PPT 編輯 huashu html2pptx 真 text frames + 編輯版 PPTX baransu 用 html2pptx 已可產生編輯版 PPTX，但底層仍是 HTML → DOM → PPTX 轉換，不是原生 Office API。要做原生需綁定 Office 365 / Keynote / LibreOffice runtime，超出 plugin 邊界。
動畫 / 影音輸出 huashu MP4 (25/60fps) + GIF + BGM TTS pipeline baransu 是靜態文件 plugin，無 frame-by-frame 渲染能力。要做需接 ffmpeg / Remotion / Lottie 等 video runtime + 音軌處理，且需要設計影片 timeline（不是 baransu 的領域）。
裝置邊框 mockup huashu iOS / Android / macOS / Browser frame components baransu 沒接 React component runtime；要做 pixel-accurate iPhone 15 Pro body（Dynamic Island / status bar）需獨立 component library + 設備 spec 維護，與設計系統職責正交。
AI 圖像生成接管 → 可達（v3.1） Kami 餵 ChatGPT Images 2.0 / huashu real image acquisition 用戶環境含 Codex（OpenAI Codex CLI），理論已支援 GPT image 生成——K3 export-brief 從「準備好給用戶手動餵 ChatGPT」升級為「可作為 Codex 端 image-gen 流程的 inline prompt brief」。paradigm 從跨工具手動升級為同環境內調用。K3 priority 因此提前。
品牌資產自動採集 → 部分可達（v3.1） huashu Core Asset Protocol 5 步 ask → search → download → verify → freeze v3.1 修訂：Core Asset Protocol 重定義為 ask → search **OR Codex image-gen** → verify → freeze。image acquisition 從「需接外部 MCP / search」變為「可走 Codex GPT image inline 生成」。J 工程量從「大」降為「中」。剩餘限制：vision-capable verify 仍需擴展。
單一視覺體系深度 Kami 一套到底（暖紙 + ink-blue + serif 無變體） baransu 設計上做三 preset（kami / google-design / swiss），平行視覺世界互斥。要做「單一視覺深度」必須砍掉兩個 preset——但這違反 baransu 「多 preset 可選」的設計選擇。做廣三個 preset vs 做深一個 preset 在 baransu 已選邊站。
Live preview / hot reload —（不在 baseline 但常見期待） baransu 是 Claude Code 內 skill，輸出靜態檔。real-time preview 需 dev server + watcher，超出 skill execution model（每次 invoke 是 one-shot）。
互動 dashboard —（KD#5 dashboard.html 靜態 SVG 的反面） v1.3 KD#5 明文：dashboard.html 為靜態 SVG mockup，不引入 chart.js / D3。理由：保持 offline-first + 無 JS 依賴 + 對應 Kami「印刷物件」哲學（紙不互動）。這是設計選擇而非能力缺陷。
本質限制總結：baransu 是「文字 / HTML pipeline 的設計子模組」，**不是 design tool 替代品**。它能讓 Claude Code 內輸出「不像 AI 做的」紙質感印刷物與 slide HTML，但不取代 Figma / Sketch / Keynote / Premiere 的角色。
v3.1 補充：用戶環境含 Codex CLI → AI 圖像生成接管 + 品牌資產採集兩條從「不可達」升級為「可達」或「部分可達」。剩下真正不可達的：原生 PPT 編輯（需 Office runtime）/ 動畫影音輸出（需 ffmpeg pipeline）/ 裝置邊框 mockup（需 component runtime）/ 單一視覺體系深度（設計選擇上 baransu 走多 preset）/ Live preview / 互動 dashboard（design choice 排除）。
