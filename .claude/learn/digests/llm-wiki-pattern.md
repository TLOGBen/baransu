# LLM 個人知識庫建構模式（LLM Wiki Pattern）

本文件是對 Karpathy 的 LLM Wiki 與 rohitg00 的 LLM Wiki v2 兩份資料的綜合精煉，涵蓋從核心概念到進階實踐的完整知識圖譜。

---

## §1 核心思想：從檢索到累積的典範轉移

大多數人使用 LLM 處理文件的方式是 RAG（檢索增強生成）：上傳文件集合，LLM 在查詢時檢索相關片段並生成答案。這個模式有其侷限——LLM 在每次查詢時都從零開始重新推導知識，沒有任何積累。要回答需要綜合五份文件的細微問題，LLM 每次都得重新找尋並拼湊相關片段，知識不會沉澱。NotebookLM、ChatGPT 檔案上傳和大多數 RAG 系統都是這種運作方式。

LLM Wiki Pattern 的核心思想截然不同：LLM **增量建立並維護一個持久的 wiki**——一套結構化、相互連結的 Markdown 文件集，位於你和原始資料來源之間。當你加入新的資料來源時，LLM 不只是建立索引供日後檢索，而是閱讀它、提取關鍵資訊，並將其整合進現有的 wiki——更新實體頁面、修訂主題摘要、標記新資料與舊主張的矛盾、強化或挑戰既有的綜合論述。知識只需編譯一次，然後**持續更新**，而非在每次查詢時重新推導。

rohitg00 的 v2 進一步強調：這個典範轉移的精髓是「**停止重新推導，開始編譯**」。RAG 檢索後即遺忘；wiki 積累並複利增長。每一次新增資料、每一次提問探索，都在使知識庫變得更豐富、更有價值。這個模式從 Vannevar Bush 在 1945 年提出的 Memex 概念汲取靈感——個人化、積極策劃、文件間的連結本身與文件同等重要。Bush 無法解決的問題是誰來做維護工作；LLM 解決了這個問題。

---

## §2 三層架構

LLM Wiki 的架構由三層組成，各自扮演不同角色，分工明確。

**第一層：原始資料（Raw Sources）** 是你精選的來源文件集合，包含文章、論文、圖片、資料檔案。這些資料是**不可變的**——LLM 只讀取它們，從不修改。原始資料是你的事實根據，是整個知識庫的基礎。

**第二層：wiki** 是 LLM 生成的 Markdown 文件目錄，包含摘要、實體頁面、概念頁面、比較分析、概覽與綜合論述。LLM **完全擁有**這一層：它建立頁面、在新資料來源加入時更新頁面、維護交叉引用、保持一致性。你負責閱讀；LLM 負責撰寫。

**第三層：schema** 是一份設定文件（在 Claude Code 中為 CLAUDE.md，在 Codex 中為 AGENTS.md），告訴 LLM wiki 的結構、約定是什麼，以及在攝取資料來源、回答問題或維護 wiki 時應遵循的工作流程。這是整個系統的關鍵設定檔——它將通用 LLM 轉化為有紀律的知識管理員，而不是普通的聊天機器人。你和 LLM 會隨著時間共同演進這份文件，逐步摸索出對你的領域最有效的做法。

rohitg00 直接點明：**schema 文件才是系統最重要的檔案**。它定義了你的領域中存在哪些類型的實體和關係、如何攝取不同類型的資料來源、何時建立新頁面而非更新現有頁面、應用哪些品質標準，以及如何處理矛盾。你和 LLM 共同演進這份文件，第一版會很粗糙，但在經歷數十個資料來源和幾次健康檢查後，你將擁有一份真正反映你的領域如何運作的 schema。

## §3 基本操作三件套（Ingest / Query / Lint）

LLM Wiki 的日常運作圍繞三個核心操作展開。

**Ingest（攝取）**：你將新的資料來源放入原始集合並告知 LLM 處理。典型流程是：LLM 閱讀來源、與你討論關鍵要點、在 wiki 中撰寫摘要頁面、更新索引、更新相關的實體和概念頁面，並在日誌中附加一條記錄。一個資料來源可能會影響 10-15 個 wiki 頁面。Karpathy 偏好一次攝取一個來源並積極參與——閱讀摘要、檢查更新、引導 LLM 強調重點；但你也可以批次攝取多個資料來源且較少監督，具體取決於你的工作風格。

**Query（查詢）**：你針對 wiki 提問。LLM 搜尋相關頁面、閱讀它們，並附引用合成答案。答案可以採用不同形式——Markdown 頁面、比較表格、幻燈片（Marp）、圖表（matplotlib）或畫布。重要洞見是：**好的答案可以歸檔回 wiki 成為新頁面**。你要求的比較、分析、發現的連結——這些都有價值，不應消失在對話歷史中。這樣你的探索就像攝取的資料來源一樣，在知識庫中複利增長。

**Lint（健康檢查）**：定期要求 LLM 對 wiki 進行健康檢查。查找頁面間的矛盾、被較新資料來源推翻的過時主張、沒有入站連結的孤立頁面、重要概念被提及但缺乏自己頁面的情況、缺失的交叉引用，以及可透過網路搜尋填補的資料缺口。LLM 擅長建議要調查的新問題和要尋找的新資料來源，讓 wiki 隨著成長保持健康。rohitg00 更進一步強調，Lint 操作不應只是建議——它應能**自動修復**力所能及的問題：孤立頁面連結或標記、過時主張標記、損壞的交叉引用修復。

## §4 索引與日誌（index.md / log.md）

兩個特殊文件幫助 LLM（以及你）在 wiki 成長時進行導航，各自扮演不同但互補的角色。

**index.md** 以內容為導向，是 wiki 中所有內容的目錄——每個頁面附帶連結、一行摘要，以及可選的日期或來源數量等元數據，按類別組織（實體、概念、來源等）。LLM 在每次攝取時更新它。回答查詢時，LLM 先閱讀索引以找到相關頁面，然後深入研究。在中等規模（約 100 個來源、數百個頁面）下，這個方法效果出奇地好，避免了基於嵌入的 RAG 基礎設施的需求。

**log.md** 以時間為導向，是事件發生和時間的仿寫記錄——攝取、查詢、健康檢查。實用技巧：如果每條記錄以一致的前綴開始（例如 `## [2026-04-02] ingest | Article Title`），日誌就可以用簡單的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 給你最後 5 條記錄。日誌讓你了解 wiki 的演進時間軸，幫助 LLM 理解近期發生了什麼。

rohitg00 補充了索引的限制：index.md 在大約 100-200 頁之前有效，超過後索引本身就變得太長，LLM 無法一次讀完，此時你需要真正的搜尋能力（詳見§7）。建議保持 index.md 作為人類可讀的目錄，但不要依賴它作為 LLM 超過約 100 頁後的主要搜尋機制。

## §5 記憶生命週期（信心評分 / 超越 / 遺忘 / 整合層級）

這是 rohitg00 v2 相對於原始模式最重要的擴展之一。原始 wiki 模式將所有 wiki 內容視為永遠同等有效。但實際上，知識有其生命週期。

**信心評分（Confidence Scoring）**：wiki 中的每個事實都應附帶信心分數——有多少來源支持它、最近一次確認是何時、是否有任何矛盾。當 LLM 寫下「Project X 使用 Redis 作為快取」時，這個主張應知道它來自兩個來源、三週前最後確認、信心值為 0.85。信心隨時間衰減，隨強化而提升。這將 wiki 從同等權重主張的平面集合，轉化為 LLM 可以說「我對 X 相當確信，但對 Y 不太確信」的活生生模型。

**超越（Supersession）**：當新資訊與現有主張矛盾或更新現有主張時，舊主張不應只是附加一條備注繼續存在。新主張應明確地取代它，帶有連結和時間戳，舊版本保留但標記為過時。這是知識的版本控制，而非只是文件的版本控制。

**遺忘（Forgetting）**：並非所有內容都應永久保存。一個永不遺忘的 wiki 會變得嘈雜。實作保留曲線：曾經重要但數月未被存取或強化的事實應逐漸淡出——不是刪除，而是降低優先級。Ebbinghaus 遺忘曲線在此效果良好：保留隨時間指數衰減，但每次強化（存取、新來源的確認）都會重置曲線。架構決策緩慢衰減；短暫性錯誤快速衰減。

**整合層級（Consolidation Tiers）**：原始觀察與已確立事實並不相同。建立一個流水線：**工作記憶**（最近的觀察，尚未處理）→ **情節記憶**（從原始觀察壓縮的會話摘要）→ **語義記憶**（從情節整合的跨會話事實）→ **程序記憶**（從重複語義中提取的工作流程和模式）。每個層級都比下一層更壓縮、更有信心、更長壽。LLM 隨著證據積累將資訊向上晉升。這就是從「我見過一次」到「這就是事物運作的方式」的路徑。

## §6 知識圖譜（實體抽取 / 有型關係 / 圖遍歷）

這是 rohitg00 v2 的另一個重要擴展：超越帶有 wikilinks 的平面頁面，在頁面之上建立有型知識圖譜。

**實體抽取（Entity Extraction）**：當 LLM 攝取資料來源時，不應只寫散文。它應提取結構化實體：人物、專案、程式庫、概念、文件、決策。每個實體獲得一個類型、屬性，以及與其他實體的關係。「React」是一個程式庫；「認證遷移」是一個專案；「Sarah」是一個擁有認證遷移並對 React 有見解的人。這種結構化提取讓知識庫遠比純散文頁面更可查詢和可導航。

**有型關係（Typed Relationships）**：並非所有連接都是平等的。「使用」、「依賴於」、「矛盾」、「導致」、「修復」、「取代」具有不同的語義權重。說「A 與 B 相關」的連結不如說「A 導致 B，由 3 個來源確認，信心 0.9」有用。類型化的關係讓知識圖譜真正有意義，而不只是一個連結清單。

**圖遍歷查詢（Graph Traversal for Queries）**：當有人詢問「升級 Redis 的影響是什麼？」時，LLM 不應只進行關鍵字搜尋。它應從 Redis 節點開始，通過「依賴於」和「使用」邊向外走，找到所有下游內容。這能發現關鍵字搜尋會遺漏的連接。知識圖譜不取代 wiki 頁面，而是增強它們：頁面用於閱讀，圖譜用於導航和發現。

## §7 可擴展搜尋（混合搜尋 BM25 + vector + 圖）

隨著 wiki 成長，搜尋能力必須跟上。原始模式依賴 index.md，在約 100-200 頁前是可行的。超過這個規模，索引本身就變得太長，需要真正的搜尋基礎設施。

**混合搜尋（Hybrid Search）**：最佳方法結合三個流：**BM25**（帶詞幹分析和同義詞擴展的關鍵字匹配）、**向量搜尋**（通過嵌入的語義相似性）、**圖遍歷**（實體感知的關係遊走）。使用倒數排名融合（Reciprocal Rank Fusion）合併結果。每個流捕獲其他流遺漏的內容：BM25 找到精確術語；向量找到語義相似性；圖譜找到結構連接。三者結合優於任何單一方法。

Karpathy 推薦 [qmd](https://github.com/tobi/qmd) 作為本地搜尋引擎，支援 Markdown 文件的混合 BM25/向量搜尋和 LLM 重排序，完全在設備上運行，提供 CLI（讓 LLM 可以調用）和 MCP 伺服器（讓 LLM 作為原生工具使用）兩種介面。你也可以自己建立更簡單的東西——LLM 可以幫助你按需編寫簡單的搜尋腳本。

在實作上，建議保留 index.md 作為人類可讀的目錄，但一旦 wiki 超過約 100 頁，就不要依賴它作為 LLM 的主要搜尋機制。混合搜尋的三個流不是全有或全無的選擇——可以從 BM25 開始，隨著 wiki 成長逐步添加向量和圖遍歷能力。

## §8 自動化：從手動到事件驅動

rohitg00 指出原始模式的最大實際缺口是：一切都是手動的。你放入資料來源並告訴 LLM 處理它；你記得定期執行 Lint；你決定何時歸檔答案。在規模化時，這個手動操作模式成為瓶頸。

理想狀態是基於事件的鉤子（hooks），自動觸發：**新資料來源時**——自動攝取、提取實體、更新圖譜、更新索引；**會話開始時**——根據最近活動從 wiki 加載相關上下文；**會話結束時**——將會話壓縮為觀察，歸檔洞見；**查詢時**——檢查答案是否值得歸檔（品質分數 > 閾值）；**記憶寫入時**——檢查與現有知識的矛盾，觸發取代機制；**按計劃**——定期 Lint、整合、保留衰減。

人類仍應參與策展和方向決策，但**書記工作——讓人們放棄 wiki 的部分——應完全自動化**。Karpathy 的原始洞見也呼應了這一點：維護知識庫的繁瑣部分不是閱讀或思考，而是書記工作——更新交叉引用、保持摘要最新、記錄新資料何時與舊主張矛盾、在數十個頁面間保持一致性。人類放棄 wiki 是因為維護負擔的增長速度快於其價值。LLM 不會感到無聊，不會忘記更新交叉引用，可以一次性處理 15 個文件。

## §9 品質與自我修正

並非所有 LLM 生成的內容品質都相同。沒有品質控制，wiki 就會積累雜訊。rohitg00 為此提出了系統性的品質保障機制。

**評分一切（Score Everything）**：LLM 寫的每一段內容都應獲得品質分數——它是否結構良好？是否引用來源？是否與 wiki 其他部分一致？你可以讓 LLM 自我評估，或使用不同提示的第二次處理。低於閾值的內容被標記為需要審查或重寫。這確保了知識庫的整體品質不會因為任何一次粗糙的攝取而下降。

**自我修復（Self-Healing）**：原始模式中的 Lint 操作應不只是建議，應能自動修復力所能及的問題。孤立頁面自動連結或標記；過時主張自動標記；損壞的交叉引用自動修復。wiki 應傾向於自動保持健康，而不只是在你記得詢問時才進行。

**矛盾解決（Contradiction Resolution）**：原始模式提到標記矛盾，但 rohitg00 指出這只是第一步。第二步是解決矛盾。LLM 應根據來源的新近度、來源的權威性和支持觀察的數量，提出哪個主張更可能正確。人類可以推翻，但預設行為通常應該是正確的。自動的矛盾解決機制讓 wiki 能夠真正反映知識的演進，而非只是積累衝突的聲明。

## §10 結晶化

Karpathy 的原始模式提到「好的答案可以歸檔回 wiki 成為新頁面」。rohitg00 將這個想法進一步發展為**結晶化（Crystallization）**的概念。

結晶化是將一個完整的工作鏈（研究線索、除錯會話、分析）自動提煉成結構化摘要的過程：問題是什麼？我們發現了什麼？涉及哪些文件/實體？浮現了哪些教訓？這個摘要成為 wiki 中的一級頁面，教訓被提取為獨立事實，強化知識庫。

你的探索本身就是一個資料來源，就像文章或論文一樣。wiki 應當如此對待它們：攝取結果、更新圖譜、強化或挑戰現有主張。每一次深入的調查，無論是程式碼除錯、競爭分析還是技術研究，都應在知識庫中留下永久的印記，成為未來探索的基石。

結晶化將知識庫從被動接收外部資料的系統，轉化為主動從內部工作中學習的系統。這實現了真正的複利效應：不只是新資料讓 wiki 更豐富，你的思考過程本身也在不斷沉澱知識。

## §11 多代理人與協作

原始模式假設單一使用者、單一代理人。許多實際使用場景涉及多個代理人或多人共同為同一知識庫做出貢獻。

**網狀同步（Mesh Sync）**：如果多個代理人並行工作（不同的程式碼會話、不同的研究線索），它們的觀察需要合併到共享的 wiki 中。最後寫入獲勝適用於大多數情況；對於衝突，基於時間戳的解決加手動推翻。這種機制確保多個來源的知識能夠一致地整合，而不是互相覆蓋或產生不可解的衝突。

**共享與私有（Shared vs. Private）**：部分知識是個人的（我的偏好、我的工作流程）；部分是共享的（專案架構、團隊決策）。wiki 需要範圍劃定。私有觀察在晉升時滾入共享知識。這個設計讓個人知識和集體知識共存，同時保持各自的完整性。

**工作協調（Work Coordination）**：當多個代理人在同一知識庫上工作時，它們需要輕量級協調——誰在做什麼、什麼被阻塞、什麼已完成。不需要完整的任務管理系統，只需足夠防止重複工作和追蹤進度。這種協調機制讓多代理人系統能夠高效協作，而不是在同一問題上浪費資源。

## §12 隱私與治理

原始模式沒有提到隱私和治理，但 rohitg00 指出這至關重要。資料來源通常包含敏感資訊：API 金鑰、憑證、私人對話、個人識別資訊（PII）。

**攝取時過濾（Filter on Ingest）**：在任何內容進入 wiki 之前，自動去除敏感資料——API 金鑰、令牌、密碼、任何標記為私有的內容。這應是自動的，而非你記得去做的事情。這個過濾層是 wiki 安全性的第一道防線，確保知識庫不會成為敏感資訊的洩漏源頭。

**稽核追蹤（Audit Trail）**：wiki 上的每一個操作（攝取、編輯、刪除、查詢）都應以時間戳記錄，包括改變了什麼和為什麼。這是你的問責層。當 wiki 中某些東西看起來不對時，稽核追蹤告訴你它是如何到達那裡的。

**大量操作的治理（Bulk Operations with Governance）**：隨著 wiki 成長，你會想要批量刪除過時內容、匯出子集，或合併重複實體。這些操作應受到稽核和可逆性保障。沒有治理的大量操作是知識庫健康的主要風險之一，一次粗心的批量刪除可能造成難以恢復的損失。

## §13 實作光譜

rohitg00 明確指出，上述所有內容都是模組化的，不需要在第一天就全部實作。根據需求選擇進入點，逐步構建能力。

**最小可行 wiki（Minimal Viable Wiki）**：原始資料 + wiki 頁面 + index.md + 描述攝取/查詢/Lint 工作流程的 schema。這大約是 Karpathy 原始模式所描述的，它有效，從這裡開始。這個起點的門檻極低，任何人都可以在幾小時內建立並開始使用。

**添加生命週期**：信心評分、取代機制、基本保留衰減。這防止 wiki 變成垃圾抽屜，讓知識庫保持新鮮和可靠。

**添加結構**：實體抽取、有型關係、知識圖譜。這讓查詢品質更好，並呈現用平面頁面會遺漏的連接。

**添加自動化**：自動攝取、自動 Lint、上下文注入的鉤子。這讓維護負擔降至接近零，wiki 真正變得自我維護。

**添加規模能力**：混合搜尋、整合層級、品質評分。這是 wiki 成長超過幾百頁時所需的能力。

**添加協作能力**：網狀同步、共享/私有範圍劃定、工作協調。這適用於團隊或多代理人設置。在每個層級上，模式都有效。不要因為沒有準備好實作所有功能而推遲開始——最小可行 wiki 本身就足夠強大。

## §14 適用場景

LLM Wiki Pattern 適用於任何需要隨時間積累知識並希望其組織而非分散的場景。Karpathy 和 rohitg00 都提供了豐富的適用場景說明。

**個人知識管理**：追蹤自己的目標、健康、心理、自我提升——歸檔日記條目、文章、播客筆記，隨時間建立結構化的自我圖景。這種應用讓個人反思從零散記錄轉化為有組織的知識庫，LLM 的維護能力在此發揮關鍵作用。

**研究與學術**：在數週或數月內深入研究某個主題——閱讀論文、文章、報告，逐步建立包含演進論文的全面 wiki。與傳統筆記不同，wiki 能主動標記矛盾、識別研究缺口，並在積累的文獻中發現新的連接。

**閱讀書籍**：逐章歸檔，建立人物、主題、情節線索及其連接的頁面。讀完時你擁有一個豐富的伴侶 wiki，類似於粉絲社群多年建立的詳細維基，但由你個人在閱讀過程中建立。

**商業與團隊**：由 LLM 維護、由 Slack 討論串、會議記錄、專案文件、客戶通話提供資料的內部 wiki，可選擇讓人類參與審查更新。wiki 保持最新，因為 LLM 承擔了團隊沒有人願意做的維護工作。

**軟體工程**：rohitg00 的 agentmemory 專案本身就是這個模式在 AI 程式設計代理人領域的具體應用。追蹤程式碼庫的架構決策、技術債務、除錯歷史和模式，讓 LLM 在未來的工作會話中能夠利用積累的上下文，而不是每次都從零開始。

**適用的通用標準**：當你在數週、數月或數年內積累相關文件或知識時；當你需要跨文件和時間的綜合而非單次查詢時；當維護負擔（保持知識更新、有組織）使你放棄手動系統時；以及當在主題上花費時間的投資值得建立持久知識資產時——LLM Wiki Pattern 都是值得採用的解決方案。
