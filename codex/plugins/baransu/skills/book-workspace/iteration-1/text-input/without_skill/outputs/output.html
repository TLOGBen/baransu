Kubernetes 三層架構總覽

Control Plane  |  Node  |  Pod  ·  核心物件與自動擴縮

架構全景圖

第一層：Control Plane（控制平面）

Control Plane 是叢集的大腦，負責全域決策（如調度）及偵測並回應叢集事件（如當 Deployment 的副本數不足時啟動新 Pod）。

核心元件
API Server
Kubernetes 控制平面的前端，所有 REST 請求的中央閘道；是唯一能讀寫 etcd 的元件。
核心元件
Scheduler
監看新建的、尚未分配 Node 的 Pod，依資源需求、親和性規則等選出最佳 Node。
核心元件
etcd
高可用分散式 KV 儲存，保存叢集所有狀態資料；所有元件皆透過 API Server 間接存取。
核心元件
Controller Manager
執行各種控制迴圈（Node controller、Replication controller 等），確保實際狀態符合期望狀態。

第二層：Node（工作節點）

每個 Node 是實際執行工作負載的機器（實體或虛擬），其上有三個必要元件共同管理容器生命週期與網路。

節點元件
kubelet
節點上的代理人，接收來自 API Server 的 PodSpec，確保容器依規格運行，並定期回報狀態。
節點元件
kube-proxy
維護每個 Node 上的 iptables / IPVS 規則，實現 Service 的虛擬 IP 流量轉發。
節點元件
Container Runtime
實際拉取映像並執行容器的引擎（containerd、CRI-O），透過 CRI 介面與 kubelet 溝通。

第三層：Pod（最小部署單位）

Pod 是 Kubernetes 的最小可部署物件，封裝一個或多個共享網路命名空間與儲存磁碟區的容器。同一 Pod 內的容器透過 localhost 互相通訊。

Pod 特性
共享網路命名空間
同一 Pod 的容器共用 IP 位址與 port 空間，透過 localhost 直接通訊。
Pod 特性
共享 Volume
Pod 內可掛載相同 Volume，讓 App 容器與 Sidecar 容器共享資料（如 log 收集）。
Pod 特性
短暫性（Ephemeral）
Pod 本身無持久性；Deployment 等控制器負責在 Pod 失敗時重新建立，確保副本數量。

核心物件（Core Objects）

Workload
Deployment
宣告式管理 ReplicaSet，確保指定數量的 Pod 副本持續運行；支援滾動更新與回滾。
Workload
ReplicaSet
由 Deployment 自動管理，負責維持 Pod 副本數量穩定；通常不直接操作。
Networking
Service
為一組 Pod 提供穩定的 ClusterIP endpoint，透過 label selector 動態追蹤後端 Pod。
Configuration
ConfigMap
以鍵值對儲存非機密設定資料，可作為環境變數或設定檔掛載進容器。
Configuration
Secret
以 Base64 編碼（可搭配 encryption at rest）儲存機密資料（密碼、Token、TLS 憑證）。

HPA — 水平 Pod 自動擴縮

HorizontalPodAutoscaler 依 CPU / Memory 使用率（或自訂 metrics）動態調整 Deployment / ReplicaSet 的副本數量，在負載升高時向外擴（scale out），負載下降時縮回（scale in），無需人工介入。

Ingress — L7 負載均衡入口

Ingress 是叢集對外暴露 HTTP/HTTPS 服務的 L7 閘道，負責 TLS 終止與基於域名 / 路徑的路由。實際執行由 Ingress Controller（如 nginx-ingress、Traefik）負責。

路徑規則 目標 Service 說明
/api/* svc-api:80 後端 API 服務，轉發至 api Pod 的 :8080
/web/* svc-web:80 前端靜態站，轉發至 web Pod 的 :3000
/（預設） svc-web:80 未匹配路徑一律導向前端
